Alexander Täuber

1 About

This is the protocol for the collection and processing of the structural data of the tree species quartets (TSQs) collected via personal laser scanning (PLS). This data has been collected in the summers of 2023 and 2024 by Tama Ray, Tobias Käcks, and Alexander Täuber. The structural data analysis including this repository and documentation was created by Alexander Täuber. In case of questions please email me, or use this repositories Issues.

This website consists of three pages which you can access with the navigation pane at the top: - The page you are on right now is a detailed description of the data collection process in the field - The Analysis Pipeline page presents the structure of the {targets} pipeline used to generate the data. - The Structural Plots page is a collection of plots describing the structure of each individual TSQ.

To inspect the dataset you can open this page on this repos website and scroll down to the datatable. To download the full dataset yourself please head over to the BEFChina Data portal

2 Raw data collection

The raw data for the TSQs of Site A was collected in August and September of 2023 by Tama Ray and Tobias Käcks. The data of Site B was collected in August of 2024 by Alexander Täuber.

In total, structural is available for 129 TSQs on Site A and 131 on Site B.

2.1 General info

The data collected are point clouds gathered via light detection and ranging (LiDAR). For this a ZEB Horizon RT personal laser scanner of the manufacturer GeoSLAM was used. The device was equipped with a small action camera, which recorded video during each scan (.mp4). The video files of the scans on Site A (2023) are only partly available, as some were lost due to data corruption. Those of Site B (2024) are fully available. No registration marks were used in either year. While this could have improved the speed of the registration and made the tree/shrub labelling much easier, it also drastically reduced the time-to-scan of each TSQ (no setting up of targets). The time-to-scan was prioritized in this campaign due to limited fieldwork time.

2.2 Scanning Track

In both years the scanner was carried through the TSQ by hand as intended for a PLS. The walked route (from here on referred to as “track”) is recorded in three dimensions (3D). The track was not consistent across TSQs because terrain was partly difficult to traverse safely. Additionally it made sense to adapt the track to the canopy structure of the focal trees/shrubs in order to achieve an optimal point cloud density. The tracks are available as polygone file formats (.ply).

2.3 Compass alignment

The ZEB-Horizon does not have an internal compass. For Site A (2023) the scans were not compass-oriented. For Site B (2024) the scans have been manually compass-oriented. This was achieved by placing a compass parralelly to the PLS-scanner at the beginning of each scan, in view of the action camera. With this information the scans of Site A could be compass-oriented to a precision of roughly 10°.

3 Point cloud processing

3.1 Raw data extraction

The ZEB-Horizon is equipped with a data logger that was read out after each field day. For this and the point cloud extraction the software GeoSLAM Hub was utilized. Here, the raw scan was registered with the track information. The resulting point clouds were inspected roughly to determine if a re-scan was necessary on each afternoon/evening of the same day. In the same software also, we exported the point clouds in .laz format and the tracks in .ply format. The latter were exported with a RGB field containing info on the track time. This way each track’s direction is visible via the colour.

3.2 TSQ tree/shrub labelling

To make automatic extraction of structural parameters possible and simplify later analyses, the TSQ tree and shrub positions were labelled manually within each scan. For this the software CloudCompare was used. Specifically, we used the point clouds, tracks, and videos to manually “find” the trees and shrubs of each TSQ in the respective scan. We then selected a point at the central base of each existing tree/shrub. We also did this for dead trees or shrubs. On control TSQs without shrubs and all other TSQs a “center” point was selected on the ground, marking where a shrub would theoretically stand. In case of missing trees, no point was selected. We then saved the selected points in a seperate .txt file with the following format:

T22_0704_tree1_ElGl,-0.699999988079,0.180000007153,0.419999986887
T22_0805_tree3_CaFa,0.209999993443,1.47000002861,0.680000007153
T22_0804_tree4_ElGl,0.560000002384,0.259999990463,0.680000007153
T22_0704_shrub_DiBu,-0.070000000298,0.759999990463,0.490000009537
T22_0704_center,-0.070000000298,0.759999990463,0.490000009537
T22_0704_north,-0.070000000298,0.759999990463,0.490000009537

The “north” (or other cardinal direction) point marks a point that together with the “center” point will form a vector describing the respective compass-direction.

3.3 Digital processing of point clouds

The digital processing of point clouds was performed using an automated pipeline built with the R {targets} package. The main steps are:

  1. Parameter Setup: All processing parameters (e.g., voxel size, DTM resolution, filtering thresholds) are defined centrally for reproducibility. For the used values please refer to the _targets.R file
  2. Raw Data Listing: The pipeline automatically detects all raw point cloud (.laz) and annotation (.txt) files.
  3. Manifest Creation: A manifest table is generated, linking each scan with its metadata and annotation files.
  4. Grouping: Data are grouped by scan for parallelized processing.
  5. DTM Extraction: For each scan, a digital terrain model (DTM) is extracted using the provided parameters.
  6. Cloud Preprocessing: The raw clouds are preprocessed in the following way:
  • If applicable the clouds are North-aligned to make Aspect extraction possible
  • The clouds are clipped to the defined local neighbourhood dimensions
  • Statistical outlier filtering is performed
  • Height normalization is performed
  • Points at the ground are clipped up to a predefined height (Removing the influence of ground- and ground-vegetation points)
  • Preprocessed local neighbourhood clouds are saved
  • Then the clouds are again clipped to the TSQ dimensions and saved
  1. TSQ Cloud Preparation: Subsets of the clouds corresponding to each TSQ are extracted.
  2. Structural Computation: Structural metrics are computed for both the “ln_clouds” and “tsq_clouds”. The following parameters are extracted:
  • Effective number of layers (ENL0, 1, 2)
  • Box dimension
  • Canopy stats: Maximum, Mean, SD, CV, Gini-Index of the Canopy heights
  • Canopy openness in % For the extraction of the Canopy stats and openness the clouds are rasterized to a pre-defined cell dimension. From the highest point in each cell the stats are calculated
  1. Result Aggregation: Results are joined into a single table, exported as a CSV file.
  2. Plot Generation: Summary plots are generated for each TSQ.
  3. Automated Reporting: The entire workflow, is rendered into this Website.
